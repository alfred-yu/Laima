## 1. 产品概述

### 1.1 产品名称

**Laima（莱码）**

- "Laima"源自拉脱维亚神话中的命运女神，寓意守护代码的命运与质量。
- 中文名"莱码"谐音"来码"，传递"来写代码"的行动号召。

### 1.2 产品愿景

> **让每一行代码都值得信赖。**

打造一个 AI 原生的新一代代码托管平台，以智能代码审查为核心竞争力，在轻量部署的前提下提供企业级全功能 DevSecOps 能力，让安全合规不再是昂贵的奢侈品。

### 1.3 产品使命

- **降低代码审查门槛**：通过 AI 让代码审查更高效、更智能、更一致
- **普惠安全合规**：将安全扫描能力作为基础功能免费提供
- **轻量全功能**：以极低的资源占用提供完整的 DevSecOps 功能集
- **数据主权可控**：支持 SaaS 和自托管双模式，用户完全掌控自己的代码

### 1.4 一句话定位

**Laima 是一款 AI 原生的轻量级全功能代码托管平台，融合 Gerrit 级严格代码审查与 GitLab 级完整 DevSecOps 能力，以 Go + Rust 高性能技术栈实现 1GB RAM 即可运行的企业级平台。**

## 2. 市场分析与定位

### 2.1 市场机会

基于竞品分析，当前市场存在以下关键空白：

| 市场空白 | 竞品现状 | Laima 的机会 |
|----------|----------|-------------|
| AI 原生代码审查 | GitHub Copilot 仅有初步能力，其他平台 AI 集成有限 | 深度集成 LLM，实现全自动代码审查 |
| 轻量 + 全功能 | Gitea 轻量但功能不全；GitLab 功能全但需 4GB+ RAM | 1GB RAM 内提供接近 GitLab 的功能集 |
| 安全合规普惠化 | GitLab/GitHub 的安全扫描需付费版（$29-$59/用户/月） | 免费版即提供基础安全扫描 |
| 严格审查 + 项目管理融合 | Gerrit 审查严格但功能单一；GitLab 功能全但审查不够灵活 | 可配置审查严格度 + 完整项目管理 |
| SaaS + 自托管无缝切换 | GitLab 体验不佳；GitHub 自托管复杂 | 统一数据格式，一键迁移 |

### 2.2 竞争定位矩阵

```
                    功能丰富度
                       ▲
                       │
          GitLab ●     │     ● GitHub
                       │
                       │
          ─────────────┼─────────────► 部署轻量性
                       │
                       │
           Gerrit ●    │     ● Gitea
                       │
                       │
                    ● Laima（目标位置）
```

**Laima 的目标位置**：在功能丰富度上接近 GitLab/GitHub，在部署轻量性上接近 Gitea，形成独特的"轻量全功能"定位。

### 2.3 核心差异化策略

| # | 差异化方向 | 核心卖点 | 竞品对标 |
|---|-----------|---------|---------|
| 1 | **AI 原生代码审查** | LLM 驱动的智能审查，自动检测 Bug、安全漏洞、性能问题，生成修复建议 | GitHub Copilot（初步）、其他平台（无） |
| 2 | **灵活审查严格度** | 可配置的审查模式：宽松（类 GitHub）→ 标准（类 GitLab）→ 严格（类 Gerrit） | 无竞品提供此能力 |
| 3 | **轻量全功能** | Go + Rust 技术栈，1GB RAM 运行完整 DevSecOps 平台 | Gitea（轻量但功能不全）、GitLab（功能全但重） |
| 4 | **安全普惠** | 免费版包含 SAST、依赖扫描、密钥检测 | GitLab/GitHub（付费版才有） |
| 5 | **开发者体验** | 现代化 UI、智能工作流、上下文感知界面 | 各平台均有改进空间 |

| 分支管理 | 分支创建、保护规则、默认分支设置、批量操作 | P0 |
| 标签管理 | 轻量标签、附注标签、签名标签 | P0 |

#### 4.2.2 Pull Request 代码审查（P0 - 核心差异化）

##### 基础审查功能

| 功能 | 描述 | 优先级 |
|------|------|--------|
| Pull Request 创建 | 支持从分支、Issue、模板创建 PR | P0 |
| 内联评论 | 行级评论、文件级评论、建议式修改（suggestion） | P0 |
| 审批规则 | 可配置审批人数、角色要求、CODEOWNERS | P0 |
| 合并策略 | Merge Commit、Squash、Rebase，可按分支配置 | P0 |
| Draft PR | 草稿模式，标记为"工作中" | P0 |
| PR 模板 | 自定义 PR 描述模板 | P1 |
| 合并冲突在线解决 | Web 端解决简单冲突 | P1 |
| Cherry-pick / Revert | 从 PR 直接 cherry-pick 或 revert | P1 |
| PR 依赖关系 | PR 之间的依赖和阻塞关系 | P2 |

##### 🌟 核心差异化：AI 智能代码审查

| 功能 | 描述 | 优先级 |
|------|------|--------|
| **AI 自动审查** | PR 提交后自动触发 AI 审查，生成结构化审查意见 | P0 |
| **Bug 检测** | AI 检测潜在逻辑错误、空指针、边界条件等问题 | P0 |
| **安全漏洞检测** | AI 检测 SQL 注入、XSS、硬编码密钥等安全问题 | P0 |
| **性能问题建议** | AI 识别性能瓶颈，提供优化建议 | P1 |
| **代码风格检查** | AI 检查代码风格一致性，建议改进 | P1 |
| **最佳实践推荐** | AI 基于语言/框架推荐最佳实践 | P1 |
| **自动修复建议** | AI 不仅指出问题，还提供一键修复补丁 | P1 |
| **审查摘要生成** | AI 为每个 PR 生成变更摘要，帮助审查者快速理解 | P0 |
| **审查意见学习** | AI 从历史审查记录中学习团队偏好，持续优化 | P2 |
| **多模型支持** | 支持接入多种 LLM（OpenAI、本地模型、国产模型） | P1 |

##### 🌟 核心差异化：灵活审查严格度

| 审查模式 | 描述 | 适用场景 |
|----------|------|----------|
| **宽松模式（Relaxed）** | 类似 GitHub，PR 创建后任何人可评论，审批规则可选 | 开源项目、小团队 |
| **标准模式（Standard）** | 类似 GitLab，至少 1 人审批，CI 通过后可合并 | 一般企业团队 |
| **严格模式（Strict）** | 类似 Gerrit，必须 2 人 +2 审批，CI 全部通过，不允许直接 push | 金融、安全、关键系统 |

**配置粒度**：
- 全局默认模式
- 组织级模式
- 项目级模式
- 分支级模式（如 main 分支用严格模式，feature 分支用宽松模式）

## 5. 技术架构设计

### 5.1 技术选型

| 层级 | 技术选型 | 选型理由 |
|------|---------|---------|
| **后端语言** | Go（主服务） + Rust（AI/安全扫描引擎） | Go：高性能、低资源占用、编译为单一二进制、部署简单；Rust：内存安全、适合 CPU 密集型任务（安全扫描） |
| **前端框架** | Vue 3 + TypeScript + Vite | 现代化开发体验、优秀的 TypeScript 支持、组件化开发 |
| **UI 组件库** | 自研组件库（基于 Radix Vue） | 打造独特品牌风格，避免与竞品同质化 |
| **代码编辑器** | Monaco Editor | VS Code 同款编辑器，功能强大 |
| **数据库** | PostgreSQL（主库） + Redis（缓存/队列） | PostgreSQL 功能强大、扩展性好；Redis 高性能缓存 |
| **对象存储** | MinIO（自托管）/ S3 兼容 API | Git LFS、制品、附件存储 |
| **搜索引擎** | Meilisearch | 轻量级全文搜索，资源占用极低 |
| **Git 服务** | go-git（Go 原生 Git 库） | 纯 Go 实现，无需系统 Git 依赖 |
| **消息队列** | 内置 Go channel + Redis Stream | 轻量级，避免引入额外中间件 |
| **容器运行时** | OCI 兼容（containerd） | CI/CD Runner 执行环境 |
| **AI 推理** | 本地模型（Ollama）+ 云端 API（OpenAI/国产模型） | 灵活接入，支持离线部署 |

## 7. 安全与合规设计

### 7.1 安全架构

| 层级 | 措施 |
|------|------|
| **传输安全** | 全站 HTTPS/TLS 1.3，SSH 密钥认证 |
| **存储安全** | 密码 bcrypt 哈希、敏感数据 AES-256 加密、数据库加密 |
| **访问控制** | RBAC + ABAC 混合模型、最小权限原则 |
| **审计追踪** | 全操作审计日志、不可篡改 |
| **输入验证** | 参数化查询、XSS/CSRF 防护、文件上传校验 |
| **密钥管理** | 密钥轮换、密钥隔离、HSM 支持（企业版） |

### 7.2 合规认证路线

| 认证 | 目标时间 | 说明 |
|------|---------|------|
| 等保二级 | v0.5 | 满足国内基础合规要求 |
| 等保三级 | v1.0 | 满足金融、政府等高合规要求 |
| SOC 2 Type II | v1.5 | 国际安全合规认证 |
| ISO 27001 | v2.0 | 信息安全管理体系认证 |

### 7.3 数据安全

- **加密存储**：所有敏感数据（密码、Token、密钥）加密存储
- **加密传输**：全站 TLS 1.3
- **备份策略**：自动增量备份 + 定期全量备份，支持异地备份
- **数据擦除**：仓库删除后安全擦除，不可恢复
- **数据导出**：支持用户一键导出所有数据（GDPR 合规）

## 12. 风险评估与应对

### 12.1 风险矩阵

| 风险 | 可能性 | 影响 | 应对策略 |
|------|--------|------|----------|
| **GitHub Copilot 持续强化 AI 能力** | 高 | 高 | 聚焦"AI 代码审查"而非"AI 代码生成"，差异化竞争 |
| **GitLab CE 功能持续增强** | 中 | 中 | 保持轻量优势，在部署体验上拉开差距 |
| **AI 审查准确性不足** | 中 | 高 | 采用"AI 辅助 + 人工确认"模式，不替代人工审查 |
| **开源社区增长缓慢** | 中 | 中 | 通过差异化功能（AI 审查、安全普惠）吸引贡献者 |
| **LLM 成本过高** | 中 | 中 | 支持本地模型（Ollama），降低云端 API 依赖 |
| **数据安全事件** | 低 | 极高 | 安全开发生命周期、定期渗透测试、Bug Bounty |
| **竞品抄袭核心功能** | 高 | 中 | 快速迭代，建立品牌和社区壁垒 |

### 12.2 应对策略详解

#### AI 审查准确性风险

- **短期**：AI 审查意见标记为"建议"而非"强制"，由开发者决定是否采纳
- **中期**：引入反馈机制，AI 从用户的采纳/忽略行为中学习优化
- **长期**：建立 AI 审查基准测试集，持续评估和提升准确率

#### LLM 成本控制

- **免费版**：提供基础 AI 审查配额（500 次/月），超出后降级为规则引擎审查
- **自托管**：推荐使用本地 LLM（Ollama + CodeLlama/Qwen-Coder），零 API 成本
- **企业版**：支持私有化部署 LLM，完全离线运行
